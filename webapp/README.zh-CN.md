# qbit-swap web app

一个用于 BTC⇄QBT 原子兑换的**与钱包无关、非托管**浏览器应用。浏览器为每笔兑换生成
临时密钥，完全在页面内签署自己的领取/退款交易（Qbit 环节使用 WASM SLH-DSA，比特币环节使用
ECDSA），并与一个仅负责转发和广播的**无密钥协调器**通信。
该应用从不询问你使用哪个钱包——你粘贴任意类型的接收地址，然后从任何你喜欢的地方发送资金。

## How it works
- 界面是一个**每屏一个决策的向导**。第一屏是唯一真正的选择——*你想以哪个方向兑换？*——因此**任何一方都可以发起**（`btc2qbt` 或 `qbt2btc`）。
- 创建者是**发起方**（持有秘密）：为自己发送的环节注资（较长的时间锁），并领取自己接收的环节（从而揭示原像）。加入者是**参与方**：为自己发送的环节注资，并用此时已公开的原像领取另一环节。
- 每一方只输入自己需要的地址（一个接收地址，以及一个仅在取消时使用的退款地址）；应用在客户端将它们解码为 scriptPubKey（`addressToScriptPubKey` 处理
  P2WPKH/P2WSH、P2TR、qbit P2MR 以及传统的 P2PKH/P2SH）。
- 应用根据协调器的实时数据流（SSE）驱动兑换，并**自动签署**领取/退款。如果兑换在时间锁到期后仍然停滞，每一方会自动退款其自己的存款。
- **看守塔安全网（默认开启）：** 一旦两个环节都已注资，应用便预先签署一组手续费阶梯式的领取交易
  以及一笔退款交易并上传，这样即使你关闭标签页，协调器也能完成（或退款）该兑换。在其武装就绪之前，应用会警告你不要离开（`beforeunload`）——只会支付到你自己的地址，
  非托管。由 `test/watchtower.e2e.mjs` 验证。
- 一个**实时在线状态指示器**显示你的对手方是否在线（依据其 SSE/API 活动）。
- 界面通过 `src/i18n.js` 提供双语支持（**English / 简体中文**）——顶栏切换器，选择会被持久化保存。

## Order book (feature-flagged, default OFF)
这里有一个可选的做市方/吃单方**订单簿**：做市方发布报价，落地页根据你选择的方向展示相应的一侧，
你点击其中一个即可买入/卖出（它会接受该报价并运行吃单方向导）。
它由 `window.QBIT_ORDERBOOK` 门控（默认关闭）——关闭时，选择一个方向会
直接进入点对点流程，这也是主要的使用体验。参见 `deploy/maker-trial.js`，这是一个
在 regtest 上发布卖单并履行吃单的做市方示例。

## Instant swap widget（功能开关，默认关闭）
落地页首屏上一个 Uniswap 风格的**一键兑换卡片**，价格来自做市机器人实时流动性（协调器的 `/rfq`
API）：选择方向，在任一侧输入金额，最优实时价格会自动填充另一侧，一键即进入常规的非托管流程
（地址 → 备份 → 实时兑换），由报价最优的做市方作为对手方。报价具有限价保护——若在报价与确认之间
价格发生对您不利的变动，会先向您展示新价格并请您确认，绝不会以更差的价格成交。由
`window.QBIT_RFQ` 门控（默认关闭）；当协调器未配置做市商时组件也会自动隐藏。参考做市机器人见
`deploy/rfq-maker-trial.js`。

## Key protection (`src/keystore.js`)
目前（出于产品决策）临时密钥以**明文**形式保存：一份用户下载的明文备份文件，
外加一份存放在 IndexedDB 中的明文副本，以便重新加载后可以恢复。一份文件保护一笔
短暂的兑换；丢失它至多只会损失那一笔兑换（临时密钥，无共享种子），而停滞的兑换仍然会退款。将该文件拖回应用即可完成或退款；结算后该记录会被清除。*（一个加密封套版本——密码短语 + WebAuthn-PRF 通行密钥，`src/passkey.js`——保留在 git 历史中，可以重新启用以提供静态保护。）*

## Build & serve
```sh
npm install          # pulls @qbit-swap/client (file:../js)
npm run build        # -> dist/app.js (single self-contained module; WASM embedded, no external fetch)
python3 -m http.server 8080    # or any static server; open http://localhost:8080
```
该打包产物面向浏览器，因此 esbuild 会自动将 Node 版 WASM 签名器替换为 `signer.web.js`。
在创建表单中（或通过 `window.QBIT_COORDINATOR`）设置协调器 URL。

## Tests
- `npm test` — 无头运行的密钥库往返测试（明文备份 + 保险库保存/列出/清除）。
- `DEV_CONFS_CAP=2 node test/bidir.e2e.mjs`（需要 regtest 实验环境）——驱动真实的 `SwapClient`
  在**两个方向上**完整走完一笔兑换，针对实时协调器和链达到 COMPLETE。
- `TRIAL_URL=https://<host> node test/browser.e2e.mjs`（需要一个正在运行的试用环境）——Playwright 驱动
  实际的向导界面（创建者 + 参与方，两个方向），通过水龙头达到 COMPLETE。
- `DEV_CONFS_CAP=2 node test/premature.e2e.mjs`（需要 regtest 实验环境）——每一方都尝试在成熟之前广播
  一笔清扫交易：受 CLTV 锁定的退款交易在链达到时间锁之前会被（节点和
  协调器）拒绝为 `non-final`，而一笔领取交易（无时间锁）会被立即接受。

## Security notes
- 密钥永不离开浏览器；协调器是无密钥的，只能看到公开数据（原像也仅在其上链之后才可见）。
- **HTLC 验证：** 在显示存款地址之前，客户端会独立地从自己的密钥 + 对手方公钥 + `H` + 锁定时间重新推导
  出两个 HTLC 的 scriptPubKey，并检查它们是否与协调器的相匹配——任何产生了无法领取/退款脚本的推导错误（或篡改）都会在任何资金转移之前中止该兑换（`SwapClient.verifyHtlc`）。
- **地址校验：** 接收/退款地址会被检查为*在正确的链上*（`addressCoin`）
  **且在正确的网络上**（`addressOnNetwork`——部署时设置 `window.QBIT_HRPS`，例如主网为
  `{btc:"bc",qbit:"qb"}`）的有效地址。因此在需要 QBT 地址的地方不能输入 BTC 地址，**并且**
  粘贴到主网兑换中的 testnet/regtest 地址会被拒绝——否则任何一种情况都会将资金发送到
  该兑换无法花费的输出。
- **注资不足：** 协调器只有在存款达到约定金额后才将某一环节计为已注资，
  因此对手方无法给其 HTLC 注资不足从而克扣另一方。
- **单次使用链接：** 第一个加入交易链接的一方占据参与方名额；第二个
  打开同一链接的人会被拒绝，而不是竞争取代对方。
- 一份备份文件保护一笔兑换；在结算前丢失它至多只会损失那一笔兑换
  （临时密钥，无共享种子）——而停滞的兑换会退款。目前备份为明文。

### Trust assumptions
- **协调器被信任会诚实地转发对手方的公钥。** 客户端会验证每个
  HTLC 都由其*自己的*密钥构建，但它接受协调器所报告的*对手方*公钥。
  因此，恶意运营者可能对公钥交换进行中间人攻击（MITM），并在原像
  揭示之后行窃。这超出了当前模型（诚实但无密钥的协调器）的范围；它**不**
  适用于协调器 bug——那些由上述 HTLC 自我验证所捕获。要完全堵住这一漏洞
  需要一个双方在注资前相互比对的带外 SAS/指纹（类似 Signal 的安全
  号码）。尚未实现。
- **时间锁** 目前为 `+20/+40` 个区块；在上主网之前，必须按每条链的实际时钟进行调整
  （BTC 约 10 分钟 vs QBT 约 60 秒的出块时间），以使领取窗口能够挺过手续费飙升 + 几轮 RBF。
